# Codex Usage Widget v0.2 GitHub Beta Release Design

## Summary

Codex Usage Widget v0.2 turns the current working prototype into a Windows
application that other users can install from GitHub Releases and keep updated.
The release remains deliberately narrow: one local Codex usage widget, no
notifications, no account system, and no heavy analytics.

The first public build supports Windows 10 and Windows 11 on x64 only. It is
distributed as an unsigned NSIS installer. Windows SmartScreen warnings are an
accepted Beta limitation and must be documented clearly.

## Goals

- Let a user download and install the widget from a GitHub Release without
  installing Node.js, Rust, or development tools.
- Build releases reproducibly in GitHub Actions from version tags.
- Check for new versions at startup and install only after user confirmation.
- Keep the widget reliable across restarts, duplicate launches, display
  changes, malformed session events, and temporary scan failures.
- Give users enough local diagnostics to report problems without exposing
  Codex prompts or session contents.

## Non-Goals

- Windows ARM64, x86, MSI, Microsoft Store, macOS, or Linux packages.
- Windows Authenticode code signing for the Beta release.
- Silent or forced updates.
- Usage threshold notifications or reset notifications.
- New charts, project attribution, repository attribution, or detailed token
  analytics.
- Telemetry, crash uploads, cloud logging, accounts, or remote storage.

## Release And Distribution

### Release Trigger

A semantic version tag triggers the release workflow. A prerelease tag such as
`v0.2.0-rc.1` creates a GitHub prerelease, while `v0.2.0` creates the public
Beta release. Normal branches and pull requests run validation but do not
publish installers.

The application version must match in:

- `package.json`
- `src-tauri/Cargo.toml`
- `src-tauri/tauri.conf.json`

The release workflow fails when these versions differ from each other or from
the tag.

### GitHub Actions

The repository will contain two workflows:

1. `ci.yml` runs on pull requests and pushes to `main`.
2. `release.yml` runs only for tags matching `v*`.

CI runs:

- `npm ci`
- frontend tests
- frontend production build
- Rust formatting check
- Rust tests
- Rust compile check

The release workflow repeats the required validation on a Windows x64 runner,
then uses the official Tauri Action to build and upload the NSIS installer and
updater artifacts to a GitHub Release.

### Release Assets

Each public release contains:

- A Windows x64 NSIS setup executable.
- The Tauri updater artifact and signature.
- Updater metadata generated for the release.
- A SHA-256 checksum file for the installer.
- Release notes with supported systems, installation steps, update behavior,
  SmartScreen guidance, and known limitations.

The NSIS installer is the only user-facing installer. MSI output is disabled to
avoid duplicate choices and an untested packaging path.

## Update Experience

The official Tauri updater plugin checks the GitHub Release update endpoint
after the main window has loaded. Update artifacts are signed with a Tauri
updater signing key stored in GitHub Actions secrets. This signing mechanism is
required for updater integrity and is separate from Windows Authenticode code
signing. The updater public key is embedded in the application configuration;
the private key never enters the repository. The endpoint uses the
`latest.json` metadata generated by Tauri Action and attached to the latest
non-prerelease GitHub Release.

Behavior:

1. Startup never blocks on the network.
2. The updater checks once per application launch after a short delay.
3. If no update is available, the UI remains unchanged.
4. If an update is available, the user sees the version and release notes.
5. The user chooses `Update now` or `Later`.
6. `Update now` downloads and installs the signed artifact, then restarts the
   application.
7. `Later` dismisses the prompt for the current launch.
8. Network and update-server errors are logged but do not interrupt local
   usage monitoring.

There is no automatic background install, forced restart, or repeated prompt
during one launch.

## Application Lifecycle

### Single Instance

The official Tauri single-instance plugin prevents duplicate background
processes. A second launch focuses and shows the existing main window.

### Tray And Close Behavior

- Closing the window hides it to the tray.
- `Show` restores and focuses it.
- `Hide` hides it.
- `Refresh` performs a real local scan.
- `Diagnostics` opens a modal panel inside the existing main window.
- `Exit` is the only tray action that terminates the process.

The application must not disappear into an unreachable hidden state. If tray
creation fails, closing the window exits normally.

### Autostart

The official Tauri autostart plugin provides a `Launch at startup` setting.
It is disabled by default. Enabling or disabling it updates Windows startup
registration immediately and the UI reflects the operating system's actual
state.

Autostart launches the normal widget window. A future start-hidden option is
outside v0.2.

### Window State

The application persists:

- Window x and y position.
- Window width and height.
- Always-on-top state.

State is saved after debounced move and resize events and when the application
exits cleanly. On launch, the saved bounds are restored only if they intersect
a currently available monitor. Otherwise, the widget uses its default
bottom-right placement on the primary monitor.

The existing minimum size remains enforced. Persisted values are validated
before use so corrupt configuration cannot create an invisible or unusable
window.

## Scanner Reliability And Schema Compatibility

The local source remains `%USERPROFILE%\.codex\sessions`, with the existing
custom path setting retained.

Scanner rules:

- Unknown JSON fields are ignored.
- Unknown event types are ignored.
- A malformed JSONL line is recorded and skipped.
- A malformed file does not stop other files from being scanned.
- A temporary path, permission, or file-sharing error does not erase the last
  valid dashboard state.
- Repeated scans remain idempotent through processed-event deduplication.
- Parsing counters distinguish files scanned, events accepted, events skipped,
  malformed lines, and I/O failures.

Rate-limit states are distinguished as:

- Available.
- No non-null rate-limit snapshot found.
- Snapshot present but unusual.
- Sessions path missing.
- Sessions path inaccessible.
- Scan failed while previously cached data remains available.

The UI continues displaying the latest valid cached values when possible, with
a warning that explains why fresh data could not be obtained.

## Database Evolution

SQLite schema changes use an explicit integer schema version stored with the
database. Migrations are transactional and applied in order at application
startup.

Rules:

- Existing v0.1 data is preserved.
- A migration failure prevents further writes and surfaces a diagnostic error.
- The application does not silently delete or recreate a user's database.
- Tests cover migration from the v0.1 schema and opening an already-current
  database.

Window and application preferences may continue using the existing
`app_config` table so there is one source of truth for local settings.

## Logging And Diagnostics

The official Tauri log plugin writes rotating logs to the platform application
log directory. Logs include:

- Application version and operating system architecture.
- Resolved sessions path.
- Database open and migration results.
- Scan start, completion, duration, and aggregate counters.
- Watcher and polling errors.
- Tray, autostart, updater, and window-state errors.
- Panic information when it can be captured.

Logs must not include raw JSONL lines, prompt text, message content, API keys,
environment dumps, or full session payloads.

### Diagnostics View

A compact diagnostics panel is reachable from the tray. It uses the existing
main window rather than creating a second application window. It shows:

- Application version.
- Supported platform and architecture.
- Sessions path and whether it exists and is readable.
- Database path.
- Log directory.
- Last scan start, completion, duration, and result.
- Last successful data update.
- Current watcher and polling status.
- Counts from the last scan.
- Last non-sensitive error.

Actions:

- `Refresh diagnostics`
- `Open logs folder`
- `Copy diagnostic summary`

The copied summary contains only the displayed non-sensitive metadata. It does
not attach logs or session files automatically.

## Public Contracts

Existing dashboard commands remain stable:

- `get_dashboard_state()`
- `refresh_now()`
- `set_always_on_top(enabled)`
- `get_settings()`
- `update_settings(settings)`

New backend contracts:

- `get_diagnostics() -> DiagnosticsState`
- `open_logs_folder()`
- `get_autostart_enabled() -> bool`
- `set_autostart_enabled(enabled: bool)`
- Window-state persistence is handled by the backend lifecycle rather than a
  general-purpose frontend file API.

`DiagnosticsState` contains only sanitized metadata and aggregate counters.

## Error Handling

- User-facing errors use short, actionable English messages.
- Recoverable background failures never block window interaction.
- Refresh buttons expose an in-progress state and cannot start concurrent
  scans.
- Updater, autostart, tray, and scanner errors are isolated from one another.
- A backend panic is logged where possible; the next launch can still open the
  existing database and report the prior failure if a panic marker exists.

## Testing

### Frontend

- Update prompt states: available, downloading, failed, dismissed.
- Refresh in-progress and failure states.
- Diagnostics rendering with missing and partial data.
- Autostart control reflects backend state and handles errors.
- Existing dashboard and heatmap tests remain green.

### Rust

- Unknown and malformed session events are skipped safely.
- Scan counters and error classifications are accurate.
- Cached dashboard data survives a failed refresh.
- v0.1 database migration preserves usage and limit data.
- Window bounds validation handles disconnected monitors and corrupt values.
- Diagnostic summaries exclude session payloads and other sensitive fields.

### CI And Packaging

- Version consistency script rejects mismatched versions.
- Pull-request CI runs on Windows.
- A release-candidate tag builds a working x64 NSIS installer.
- The installed application starts without development dependencies.
- The updater accepts a correctly signed newer release and rejects an invalid
  signature.

### Manual Acceptance

- Clean install on Windows 10 x64 and Windows 11 x64.
- SmartScreen guidance matches the actual unsigned installer flow.
- Install and upgrade preserve the application database and settings.
- Uninstall removes the application but leaves the local database, settings,
  and logs in the application data directory so reinstalling can recover them.
  The README documents how to remove these files manually.
- Duplicate launch focuses the existing widget.
- Close-to-tray, tray restore, refresh, diagnostics, and exit all work.
- Autostart survives reboot and can be disabled.
- Window position restores on the same monitor and recovers after monitor
  removal.
- Missing, inaccessible, empty, and malformed sessions directories do not
  crash the widget.

## Documentation

The README will include:

- A prominent Windows x64 download link.
- Product screenshot.
- Installation and uninstall instructions.
- SmartScreen explanation for the unsigned Beta.
- Local-data and privacy statement.
- Automatic update behavior.
- Supported Codex log source and known schema-compatibility limitation.
- Troubleshooting steps and diagnostics instructions.
- Development and release commands for contributors.

The repository will also include a concise `CHANGELOG.md` and an issue template
that asks for the copied diagnostic summary rather than session files.

## Rollout

1. Land infrastructure and stability changes behind version `0.2.0`.
2. Produce a prerelease tag and validate installation and update flow on clean
   Windows 10 and Windows 11 environments.
3. Publish `v0.2.0` as a GitHub Beta release after manual acceptance.
4. Use diagnostics from early users to prioritize compatibility fixes.

## Success Criteria

v0.2 is complete when a Windows x64 user can:

- Download one NSIS installer from GitHub Releases.
- Install and run the widget without a development toolchain.
- Preserve existing local usage data during upgrade.
- Opt into autostart and reliably restore the window.
- Receive a signed in-app update prompt for a newer GitHub Release.
- Recover from common local-log failures without losing the last valid state.
- Copy a safe diagnostic summary that is useful in a GitHub issue.
